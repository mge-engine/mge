Tracing

MGE defines a tracing using a system of trace topics having active trace levels and trace sinks configured.

  • A trace level defines whether a trace message is informational or describes a severe error.

  • A trace sink is a mechanism to write trace records to - like a file or the console.

  • A trace topic is a named category of trace, which allows an active level to be set, and trace sinks attached as destination of trace records

  • A trace formatter is a mechanism to create text from a trace record.

  • A trace record is created for each trace entry, trace records capture the trace message and details on its origin.

To use tracing, mge/core/trace.hpp needs to be included.

In addition to use configuration to enable and disable trace, the following environment variables can be used:

MGE_TRACE_TO_STDOUT

Set to “1” to print all trace to standard output.

MGE_ENABLE_TRACE

Set to “1” to enable all trace.

Severity

Trace messages can have a different severities, indicated by a trace level.

Severity

Purpose

ERROR

Severe malfunction message.

WARNING

Potential problem.

INFO

General information message.

DEBUG

Detailed debug information.

FATAL

Fatal message, program will exit.

Trace Topic

Trace topics partition trace output into different categories. Trace can be enabled or disabled per category. There exists a pseudo-category for enabling and disabling trace at a global level.

Defining a Trace Topic

In a source file, a trace topic must be defined. This is best done using the predefined macro MGE_DEFINE_TRACE. This macro can be used within a namespace.

MGE_DEFINE_TRACE(TOPIC)

Define a trace topic.

Parameters:

  • TOPIC – name of trace topic

Using a Trace Topic

If you just want to use a trace topic for tracing, a forward declaration must be done. This is done using MGE_USE_TRACE, or MGE_USE_IMPORTED_TRACE in case the topic is defined in another dynamic library. These macros need to be used in the same namespace as the trace topic was defined.

MGE_USE_TRACE(TOPIC)

Use an already defined trace topic locally.

Parameters:

  • TOPIC – name of trace topic

MGE_USE_IMPORTED_TRACE(TOPIC)

Use an already defined trace from another library locally.

Parameters:

  • TOPIC – name of trace topic

To use a trace topic in a different namespace, the following macro can be used to have a using declaration for the trace topic.

As an example, writing trace into a trace topic TEST looks like this:

MGE_ERROR_TRACE(TEST) << "This is a test of error trace";

Note that a line break isn’t needed at the end.

During development one may want to add temporary debug trace, which is purely for isolating the problem, and which should be easy to identify. A MGE_XDEBUG statement may fulfil this purpose. Trace messages issued by this statement are prefixed with ‘XDEBUG:’ and thus are easy to grep for. Note that the released code of MGE does not contain any ‘XDEBUG’ trace.

MGE_XDEBUG(TOPIC)

Invoke debug trace.

Used for temporary tracing when hunting bugs or alike. Submitted changes should never have a MGE_XDEBUG trace statement.

Parameters:

  • TOPIC – trace topic
