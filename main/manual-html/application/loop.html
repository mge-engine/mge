Interaction Loop¶

Game Loop Concept¶

The interaction loop of a game application can be usually described by the following flow:

+-------------------------------------------------+
|                                                 |
+-> (!quit) -> (input) -> (update) -> (present) --+

While there is no quit signalled, the application processes the following steps in a loop:

input

Input is processed, state of input devices is retrieved.

update

The state of the application is updated.

present

The state of the application is presented at the screen.

Abstractions¶

The target of the loop is abstracted in the interface class mge::loop_target:

class loop_target¶

Loop target pattern for interactive update.

Subclassed by mge::application

Public Functions

virtual bool is_quit() const = 0¶

Retrieve whether the interactive loop shall quit.

Returns

true if the interaction loop shall quit

virtual void input(uint64_t cycle) = 0¶

Retrieve and process input in an loop step.

Parameters

cycle – frame counter

virtual void update(uint64_t cycle, double elapsed, double delta) = 0¶

Update state in a step.

Note cycle may be the same for multiple calls.

Parameters

  • cycle – frame counter

  • elapsed – total elapsed time

  • delta – delta in seconds

virtual void present(uint64_t cycle, double peek) = 0¶

Presents the state.

Parameters

  • cycle – frame counter

  • peek – percentage of peek into next step if a fixed step mechanism is used, in range [0,1).

The loop itself is implemented by implementing a component of mge::loop interface:

class loop : public mge::component<loop>¶

Interaction loop abstraction.

While the loop target does not signal quitting, input is collected from target, state is updated and presented.

Subclassed by mge::fixed_step_loop, mge::simple_loop

Simple Loop¶

A simple loop is provided which does not use fixed step update, and also does not provide any peek on presentation.:

auto simple_loop = mge::loop::create("simple");

The simple loop processes the interaction loop as follows:

  • update will be called with the delta time compared with previous frame

  • present will always be called with a peek value of 0.0.
